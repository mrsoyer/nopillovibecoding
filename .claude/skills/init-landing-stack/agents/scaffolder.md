# Agent : Scaffolder

## Mission
A partir du brief de l'interviewer, executer la phase 2 du workflow : creation du dossier projet, install des dependances, generation de tous les fichiers depuis les templates.

## Input
Le brief produit par l'interviewer (variables `{{PROJECT_NAME}}`, `{{PROJECT_TYPE}}`, etc.).

## Process

Charge [../references/scaffold-patterns.md](../references/scaffold-patterns.md) pour les commandes exactes.

### Etape 1 — Pre-requis

Verifier en bash :
```bash
node --version | grep -E "^v(2[2-9]|[3-9][0-9])\." || exit "Node 22+ requis"
command -v supabase || npm install -g supabase
command -v netlify  || npm install -g netlify-cli
```

Si erreur fatale → STOP avec instruction precise (ex: `brew install node@22`).

### Etape 2 — Dossier projet + git

```bash
PROJECT_DIR="$HOME/projets/{{PROJECT_NAME}}"   # ou ailleurs si demande
mkdir -p "$PROJECT_DIR"
cd "$PROJECT_DIR"
git init -b main
```

Demander confirmation a l'utilisateur si le dossier existe deja (proposer suffixe `-2`).

### Etape 3 — Astro + Tailwind + Supabase JS

```bash
npm create astro@latest front -- \
  --template minimal \
  --typescript strict \
  --install \
  --git no \
  --yes

cd front
npx astro add tailwind --yes
npm install @supabase/supabase-js
npx astro add react --yes    # necessaire pour ContactForm.tsx (island)
cd ..
```

### Etape 4 — Structure annexe

```bash
mkdir -p supabase/{migrations,functions/contact-form}
mkdir -p .claude/{rules,skills}
mkdir -p docs
```

### Etape 5 — Application des templates

Pour chaque template du tableau (cf. scaffold-patterns.md, section "Templates a appliquer (MVP)") :

1. Lire le fichier source `.claude/skills/init-landing-stack/assets/templates/<path>`.
2. Substituer les `{{VAR}}` connues (PROJECT_NAME, PROJECT_TYPE, TIMESTAMP). Les variables NL_/SB_ restent en placeholder, elles seront resolues apres phases 3-4.
3. Ecrire le resultat dans le projet.

Utiliser la fonction `substitute()` definie dans scaffold-patterns.md.

Order important : appliquer les fichiers config (CLAUDE.md, netlify.toml, .gitignore, README) AVANT les fichiers de code (Hero.astro, ContactForm.tsx) pour eviter des conflits si l'utilisateur a deja un fichier (cas rare apres `mkdir` mais possible).

### Etape 6 — Verification post-scaffold

Lancer les checks bash listes dans scaffold-patterns.md "Verification post-scaffold".

Si un check echoue → STOP, afficher quelle etape, proposer fix.

### Etape 7 — Commit initial

```bash
git add -A
git commit -m "chore: init landing stack (Astro 6 + Tailwind + Supabase + Netlify)

Generated by init-landing-stack skill.
- Frontend  : front/ (Astro 6 + Tailwind 4 + TypeScript strict)
- Backend   : supabase/ (migration leads + edge function contact-form)
- Context   : .claude/{rules,skills/context-guardian}
- Config    : CLAUDE.md, netlify.toml, README.md"
```

## Output

Status retourne au workflow principal :
- `success` : tous les fichiers en place, commit fait
- `partial` : certains templates non substitues (NL_/SB_ vars), c'est attendu — sera complete apres phases 3-4
- `error` : avec liste des etapes a corriger manuellement

Variables disponibles pour la suite :
- `$PROJECT_DIR` (chemin absolu du projet)
- Tous les fichiers presents

## Regles

- **Pre-requis avant scaffold** : ne pas commencer si Node < 22 ou CLIs manquantes.
- **Dossier vide** : si non vide, demander confirmation, jamais ecraser sans accord.
- **Substitution prudente** : ne pas substituer les vars qui ne sont pas encore connues. Laisser `{{NL_SITE_URL}}` en placeholder.
- **Verification systematique** apres chaque etape critique.
- **Commit Git initial** obligatoire — donne un checkpoint propre.

## Anti-patterns

- ❌ Skipper la verification post-scaffold.
- ❌ Substituer toutes les vars d'un coup (NL_/SB_ pas encore connues).
- ❌ Lancer `npm install` global sans verifier le contexte (peut polluer).
- ❌ Commit avec un message generique ("init project").
