Skill / Rule / MCP / Hook

Les 4 mecanismes pour ancrer Claude dans la realite de TON projet.

Formation Vibecoding · Jour 1 · Thomas (Feaderz)

Apres 40 min, vous saurez quand utiliser chaque mecanisme. Vous allez l'appliquer immediatement en creant VOTRE skill cet apres-midi. Insister sur la promesse : repartir avec un artefact dans .claude/skills/[nom-leur-skill]/.

Le probleme

A chaque conversation, Claude oublie :

  • Tes conventions de code
  • L'architecture de ton projet
  • Tes workflows recurrents
  • Tes regles de qualite
Consequences :
  • Tu retapes tout a chaque session
  • Resultats incoherents entre devs
  • L'IA "hallucine" faute de contexte
Question : qui a deja vu Claude faire un truc qui n'a rien a voir ? Cause #1 : il n'a pas le contexte de ton projet. Solution : 4 mecanismes pour persister le contexte. Faire lever la main : "qui a deja eu une hallucination ?" — la moitie de la salle.

Les 5 mecanismes de Claude Code

Mecanisme Quand Comment c'est charge
CLAUDE.md Regles globales projet Auto chaque session
Rule CE QUE fait UN fichier Auto si match paths:
Skill COMMENT faire UN workflow A la demande (trigger)
MCP Outils externes Au boot session
Hook Action qui DOIT s'executer Lifecycle event

Skill = competence (verbe). Rule = doc d'un fichier (nom).

Skill = competence (verbe). Rule = doc d'un fichier (nom). C'est le truc le plus important de la journee. On le repete. On va voir chacun en detail dans les 8 slides suivantes.

CLAUDE.md — le contexte permanent

  • 📄 Fichier : CLAUDE.md a la racine du projet
  • 🔄 Charge : automatiquement chaque session
  • 🎯 Contenu : conventions globales

Exemple (nopillo-landing-exemple) :

# CLAUDE.md
## Stack
- Astro 6 + Tailwind 4 + Supabase + Netlify
## Conventions
- PUBLIC_* pour env client
- Composants .astro statique, .tsx pour islands
## Cibles
- Lighthouse >= 95
- LCP < 2s
C'est la fiche d'identite de votre projet. Stack, conventions, cibles de qualite. Charge dans CHAQUE conversation, sans rien faire. Montrer le vrai CLAUDE.md du repo template s'ils l'ont sous la main.

CLAUDE.md : la regle des 80 lignes

< 80 lignes  →  Claude suit

> 80 lignes  →  Adherence chute

Si tu depasses :

  • Sortir le detail dans docs/
  • Linker depuis CLAUDE.md
  • Verifier avec /context-guardian

Source : test reel sur 100+ projets.

C'est verifie en pratique. > 80 lignes, Claude ignore. Concept : recence + primacy. Trop d'info = dilution. Si t'as 200 lignes de regles, c'est que t'as 3 rules et 1 skill cachee dedans. Demander : "qui a un CLAUDE.md > 80 lignes ?" — faire confesser.

Rules — la connaissance scopee

  • 📄 Fichier : .claude/rules/[nom].md
  • 🔄 Charge : auto si match paths:
  • 🎯 Contenu : ce que fait UN fichier / dossier
.claude/rules/
├── frontend.md     ← regles frontend globales
├── backend.md      ← regles backend globales
├── front/
│   ├── hero.md     ← regle scopee a Hero.astro
│   └── form.md     ← regle scopee a Form.tsx
└── back/
    └── contact-form.md
Une rule = doc d'un fichier specifique. Pas charge tout le temps. Seulement quand Claude touche au fichier. C'est ton historique : "qu'est-ce que fait ce composant ? Pourquoi ?". Comparer avec un Confluence ou Notion : c'est pareil mais a cote du code.

Rule : exemple concret (Hero.astro)

---
paths:
  - "front/src/components/Hero.astro"
---

# Hero — composant LCP

## Role
Premiere section visible. Element LCP du Lighthouse.

## Contraintes
- <Image> Astro, loading="eager"
- Texte CTA depuis front/.env (HERO_CTA)
- Pas d'animation framer-motion (CLS)

## Dependencies
- HeroBackground.tsx
- CTAButton.astro
paths: c'est le scope. Sans paths:, charge tout le temps = pollution. Sections classiques : Role / Contraintes / Dependencies. Quand Claude touche Hero.astro, il sait pourquoi telle decision. C'est tres utile pour onboarder un nouveau dev OU un nouvel agent.

Skills — les competences workflow

  • 📄 Fichier : .claude/skills/[verbe]/SKILL.md
  • 🔄 Charge : a la demande (trigger description)
  • 🎯 Contenu : procedure repetable avec etapes

Skill = un VERBE.

/deploy-staging      → deployer
/audit-lcp          → auditer
/clone-section        → cloner
/generate-favicon    → generer
/scan-broken-links   → scanner

"Je tape les memes etapes 3x ?" Cree un skill.

Skill = competence. Toujours un verbe. Trigger : Claude detecte un mot-cle dans ta phrase → lance le skill. C'est ce que vous allez creer cet apres-midi. Faire lister des verbes que les participants tapent souvent.

Anatomie d'un bon skill

①
name
Verbe kebab-case
②
description
"Use when X, Y, Z"
③
etapes
5-8 numerotees
④
sortie
Livrable concret 
---
name: audit-lcp
description: Audit Lighthouse + propose 3 optimisations LCP.
  Use when "audit perf", "verifier LCP", "optimiser performance".
allowed-tools: Read Write Bash
model: sonnet
---

1. cd front && npm run build → OK ?
2. npm run preview & lighthouse → score
3. Identifier 3 issues prioritaires
4. Output : docs/audit-[date].md
Description avec triggers explicites = c'est CRITIQUE. Sans "Use when X, Y, Z", Claude ne sait pas quand lancer. 5-8 etapes max. 20 = decoupe en 2 skills. Insister : les 4 elements sont OBLIGATOIRES pour leur skill cet aprem.

MCP — les outils externes

MCP = Model Context Protocol

Skill
procedure (Claude suit etapes)
MCP
outil concret (Claude appelle fonction)

Exemples installes :

  • mcp__claude_ai_supabase__* — execute SQL, deploy edge function
  • mcp__playwright__* — browser automation
  • mcp__webflow__* — Designer + Data API

Un skill peut UTILISER un MCP via allowed-tools:

MCP c'est les outils dispos pour Claude (Supabase, Webflow, Playwright). Comme des fonctions JS, mais cote serveur. Tape `claude mcp list` dans ton terminal pour voir ce qui est dispo. Demo possible : faire `claude mcp list` en live a la fin de la slide.

Hooks — l'enforcement deterministe

Quand : une action DOIT s'executer (lint, audit).

Skill
Claude decide d'appeler (peut oublier)
Hook
Le harness execute (oubli impossible) 
// .claude/settings.json
{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Edit|Write",
      "command": "cd front && npm run lint -- --fix"
    }]
  }
}

→ Apres CHAQUE Edit ou Write, lint se lance.

Hook = enforcement, garanti. Skill : Claude peut oublier. Hook : c'est le systeme qui force. Useful pour : lint apres edit, audit avant commit, format auto. Slide courte (1 min), enchainer vite sur decision tree.

Le decision tree

Q1: Procedure repetable avec etapes ? OUI → SKILL (.claude/skills/[verbe]/) NON → Q2 Q2: Fait permanent toutes sessions ? OUI → CLAUDE.md (< 80 lignes) NON → Q3 Q3: Fait specifique a UN fichier ? OUI → RULE (.claude/rules/ avec paths:) NON → Q4 Q4: Action qui DOIT s'executer ? OUI → HOOK (settings.json) NON → Q5 Q5: Outil / API externe ? OUI → MCP (claude mcp add ...) NON → Pas besoin d'ajouter de contexte
Vous avez UN doute ? Vous prenez ce tree. C'est LA slide a photographier MAINTENANT avec votre tel. On va l'appliquer sur des cas concrets a la slide suivante. Laisser 30 sec de pause pour qu'ils sortent les tels.

Ma methode : Documentation-First

Je ne code RIEN avant d'avoir saoule Claude de contexte.

Le principe :

  • Plus Claude a de contexte  →  moins il hallucine
  • Plus Claude est cadre  →  moins de bugs
  • Plus c'est documente  →  plus c'est reproductible

Mon stack mental :

1. ENQUETER  →  2. CADRER  →  3. CAPITALISER  →  4. EXECUTER

Si vous retenez UN truc de la journee : ce slide. Je ne lance JAMAIS un projet sans avoir saoule Claude de docs avant. C'est exactement ce que vous avez fait ce matin : je vous ai donne le repo + docs + rules + skills. Resultat : vous avez code direct. Sans ca : Claude invente, vous corrigez 50% du temps. INSISTER : cette slide est la signature de ma methode.

Le pipeline en pratique

① ENQUETER /doc-maker 5-10 docs en parallele
~10 min
② CADRER /cdc-maker 1 CDC + 20-30 taches
~15 min
③ CAPITALISER /skill-maker 1 skill par workflow
~20-30 min
④ EXECUTER Claude Code Contexte riche
0 hallucination
Sur ce projet de formation : 25 docs  ·  1 CDC  ·  6 skills

→ Le bug n'est plus Claude. C'est le brief.

Etape 1 : je lance 5-10 /doc-maker EN PARALLELE sur concurrents, stack, best practices, anti-patterns. Etape 2 : avec ces docs, /cdc-maker structure un CDC en 15 min. Etape 3 : je repere les workflows recurrents → 1 skill par workflow. Etape 4 : la, Claude code avec TOUT le contexte. Resultat : 0 hallucination. Question : "Combien de docs sur un nouveau projet client ? Devinez." Reponse : "Moi je tape souvent 10-15 docs avant la premiere ligne de code."

Top 7 anti-patterns a fuir

✗ CLAUDE.md > 80 lignes → Sortir le detail dans docs/
✗ Skill sans triggers explicites → "Use when X, Y, Z"
✗ Rule sans paths: → Toujours scoper
✗ Skill = "fait un truc avec X" → 1 verbe, 1 livrable
✗ Skill avec 20 etapes → Decouper en 2 skills
✗ Hook sans confirmation → Operations non destructives
✗ Secrets dans CLAUDE.md → Variables d'env + .gitignore
Ce sont les 7 trucs qui cassent le truc. Si vous en evitez 5 sur 7, vous etes deja meilleurs que 80% des devs Claude Code. Slide rapide, on enchaine sur les demos.
